\section{UML Diagrams}
\subsection{Activity Diagram}
The activity diagram illustrates the complete operational flow of the application — from user login and authentication, through face recognition, voice interaction, and emergency handling, to caregiver management and logout. It captures how the system responds to different user roles, inputs, and conditions, providing a clear and structured view of how all components and features work together in real time.

\begin{figure}[!htbp]
    \centering
    \includegraphics[width=\textwidth, angle = 270]{activity.png}
    \caption{Activity diagram}
    \end{figure}
    
\newpage
\begin{flushleft}
From Figure 4.1, we can get the following Inference:

\textbf{1. Application Start \& User Check} \\
The flow begins at \textbf{START}. The system checks whether the user is an existing user. If \textbf{No}, the user is directed to \textbf{Setup Profile} to register. If the authentication \textbf{Fails}, the setup profile step is triggered again. If \textbf{Yes}, the system proceeds.

\textbf{2. Role-Based Routing} \\
After successful authentication, a \textbf{Role?} decision node routes the user to either the \textbf{Patient Dashboard} or the \textbf{Caregiver Dashboard} based on their assigned role.

\textbf{3. Face Recognition (Patient Side)} \\
Once the patient is authenticated, the camera runs \textbf{Face Recognition}. The result is evaluated based on a \textbf{Match?} confidence level:
\begin{itemize}
    \item \textbf{High} $\rightarrow$ The identity is announced to the patient via \textbf{TTS (Text-to-Speech)}
    \item \textbf{Low} $\rightarrow$ The system displays a \textbf{Show New Face} prompt
    \item \textbf{Medium} $\rightarrow$ The face is \textbf{Queued for Verification} by the caregiver
\end{itemize}

\textbf{4. Patient Dashboard}
The patient has access to multiple features from the dashboard:
\begin{itemize}
    \item \textbf{Voice Input} $\rightarrow$ Captured and converted via \textbf{Speech-to-Text}, then the system checks the \textbf{Intent}:
    \begin{itemize}
        \item \textbf{AI Assistant} $\rightarrow$ Engages in conversation with Orito (Gemini + Sarvam)
        \item \textbf{Reminders} $\rightarrow$ Opens \textbf{Manage Reminders} for medication and events
        \item \textbf{Settings} $\rightarrow$ Opens the settings screen
    \end{itemize}
    \item \textbf{SOS Trigger} $\rightarrow$ The patient can trigger an SOS directly. This activates \textbf{Emergency SOS}, which simultaneously:
    \begin{itemize}
        \item \textbf{Notifies Caregivers} instantly
        \item \textbf{Displays} an emergency alert on screen
        \item If \textbf{No Response?} within a timeout $\rightarrow$ the system automatically \textbf{Calls 100} (emergency services)
        \item If \textbf{Yes Response} $\rightarrow$ the alert is acknowledged and resolved
    \end{itemize}
    \item \textbf{AI Reminders} $\rightarrow$ Accessible through settings for managing scheduled reminders
\end{itemize}

\textbf{5. Caregiver Dashboard}
The caregiver has a dedicated dashboard with the following capabilities:
\begin{itemize}
    \item \textbf{View Patient Status} $\rightarrow$ Monitor the patient's current condition
    \item \textbf{Manage Reminders} $\rightarrow$ Add or update medication and event schedules
    \item \textbf{Verify Faces} $\rightarrow$ Review and confirm unidentified faces queued from the patient's face recognition
    \item \textbf{View Alerts} $\rightarrow$ Check all triggered SOS and system alerts
    \item \textbf{Receive SOS} $\rightarrow$ Get notified instantly when the patient triggers an emergency
    \item \textbf{Navigate to Patient} $\rightarrow$ Access GPS location to reach the patient
    \item \textbf{Browse Memories} $\rightarrow$ View the patient's memory journal and past conversations
    \item \textbf{Memories} $\rightarrow$ Manage stored interaction logs
\end{itemize}

\textbf{6. Logout \& End}
Both the patient and caregiver flows include a \textbf{Logout?} decision. If the user chooses to logout, the session ends at the \textbf{END} node. If not, the system loops back to the respective dashboard, keeping the session active.
\end{flushleft}

\end{figure}

\newpage
\begin{flushleft}
The Figure 4.2 illustrates how all components of A.U.R.A interact across its key functionalities:
\begin{itemize}
    \item \textbf{Face Recognition:} The AURA module detects a face, sends the result to the backend, and the identified person's name is displayed to the patient.
    \item \textbf{Voice Interaction:} The patient speaks in Malayalam, which is converted to text and sent to the backend. Gemini generates a response with a distress check, and the reply is converted back to Malayalam audio and played to the patient.
    \item \textbf{Conversation Logging:} Every interaction is saved to MongoDB, maintaining a persistent memory journal for the patient.
    \item \textbf{Schedule Management:} The app communicates with the backend to add or retrieve medication and event schedules stored in MongoDB.
    \item \textbf{SOS Alert:} An emergency triggered by voice or button sends an instant alert through the backend to the caregiver.
    \item \textbf{GPS Tracking:} The app continuously sends location data to the backend, allowing caregivers to track the patient in real time.
    \item \textbf{Caregiver Authentication:} Firebase Google Sign-In verifies the caregiver's identity before granting restricted access to patient data.
    \item \textbf{Bluetooth Connection:} The mobile app discovers and connects to the local AURA module via Bluetooth to initiate the entire system.
\end{itemize}
\end{flushleft}

\end{figure}
\newpage
\begin{flushleft}
The Figure 4.4 represents the interactions between three actors --- \textbf{Patient}, \textbf{Caregiver}, and \textbf{Admin} --- within the A.U.R.A system.

The \textbf{Patient} has the most interactions, including logging in, using the voice assistant, adding event schedules, receiving event and medication reminders, confirming medication intake, saving memories, viewing the memory journal, and triggering an SOS in emergencies.

The \textbf{Caregiver} has restricted access focused on patient safety and health management, which includes viewing medication details, updating the medication schedule, and receiving emergency alerts when an SOS is triggered.

The \textbf{Admin} is connected to the Login use case, indicating a role-based access control where the admin manages system-level access separately from the patient and caregiver.

Overall, the diagram shows how A.U.R.A serves different users with role-specific features, keeping patient-facing functions separate from caregiver and admin controls.
\end{flushleft}

\section{DB Design}
\subsection{ER Diagram}
\begin{figure}[!htbp]
    \centering
    \includegraphics[width=\textwidth, angle = 270]{ER.png}
    \caption{ER diagram}
\end{figure}
\newpage
\begin{flushleft}
The Figure 4.3 represents the database structure of the A.U.R.A system, built around the central \textbf{Patient} entity.

The \textbf{Patient} (with attributes: name, email) is the core entity connected to most parts of the system. A patient \textbf{Generates} interaction logs through the \textbf{Interaction Log} entity, which records conversation content, timestamp, and LLM ID. The patient also \textbf{Takes} medications managed through the \textbf{Medication} entity, which holds details like name, dosage, and frequency. Medication schedules are further linked to a \textbf{Reminder} entity that tracks the time, status, and reminder ID via a \textbf{Scheduled For} relationship.

The patient is \textbf{Assigned To} a \textbf{Device} (with attributes: type and battery level), which is the local AURA hardware module. A \textbf{Caregiver} (with attributes: name and phone) \textbf{Monitors} the patient and is \textbf{Notified} during emergencies through the \textbf{SOS Alert} entity, which captures the GPS location, timestamp, trigger type, and SOS ID. The patient \textbf{Triggers} these SOS alerts directly.

The \textbf{Admin} entity (with admin ID and role) \textbf{Audits} the SOS alerts, providing system-level oversight and control.

Overall, the diagram shows how patient data, device assignment, medication management, conversation history, emergency alerts, and role-based access are all interconnected within A.U.R.A's database.
\end{flushleft}
